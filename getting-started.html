Nexus Start-to-End Setup Guide

This guide is for first-time users and beta testers. It covers install, license activation, LM Studio setup, task creation, and practical troubleshooting so you can get to stable, repeatable runs quickly.

1. Preflight Checklist

Before opening Nexus, verify these five items so your first run is clean:

  • You have your license key in format NEXUS-XXXX-XXXX-XXXX-XXXX.
  • Your target platforms are signed in at least once manually (LinkedIn and/or Indeed).
  • Your resume PDF is finalized and saved locally.
  • LM Studio is installed (if using local LLM mode).
  • You know your daily application limits for each platform.
Goal: get one successful end-to-end application first, then scale volume.

2. Install + Activation

  1. Download the latest Nexus build from the beta channel.
  2. Install and launch the app.
  3. Paste your license key on the activation screen and click Activate.
  4. If activation fails: verify internet connectivity and retry once.
Once activated, Nexus stores local license state and periodically verifies in background.

3. LM Studio Setup (Recommended: Qwen 2.5 7B Instruct)

Install and load model

  1. Install LM Studio from the official site.
  2. Open LM Studio, download qwen2.5-7b-instruct (or your preferred model).
  3. Load the model into memory.
  4. Start LM Studio local server.

Nexus settings

  • Provider: LM Studio
  • API URL: http://localhost:1234/v1
  • Model: leave blank to auto-select loaded model, or set explicit model ID
If model name is blank, Nexus now auto-picks the first loaded model from LM Studio.

Verify connection

  1. Click Test LLM Connection in Settings.
  2. Expected result: Connected! Model: ...
  3. If it fails, read the returned endpoint and status message carefully.

Fast sanity check in terminal

curl http://127.0.0.1:1234/v1/models

Common fix

Set URL with /v1 and reload model in LM Studio.

4. Sessions + Accounts

Use the Accounts page in Nexus to verify platform sessions before running tasks.

  1. Open LinkedIn session window and sign in.
  2. Open Indeed session window and sign in.
  3. Refresh status inside Nexus until each shows connected.
If task start says “no login detected” but dashboard looks connected, refresh session status from Dashboard once before starting.

5. Resume + Auto-Fill Defaults

  1. Upload resume in Resumes tab.
  2. Confirm file path via “Open location” when needed.
  3. Set profile defaults: location, work authorization, phone, etc.
  4. If available in your build, set demographic defaults (race, etc.) carefully.

For consistent outcomes, keep one primary resume per task and avoid switching mid-run.

6. Create Your First Task

Baseline task recipe

  1. Choose site (LinkedIn or Indeed first).
  2. Add keyword + location.
  3. Select resume for this task explicitly.
  4. Set minimum match score (example: 7.5/10).
  5. Set application limit (example: 10/day for calibration).
  6. Enable Easy Apply only for first stable runs.

Good defaults

  • Safe mode: ON
  • External apply: OFF until base flow is stable
  • Discord webhook: ON for run completion and module gaps

7. Run + Review Loop

  1. Start one task and watch first 2-3 applications closely.
  2. Confirm question answering behavior (text, dropdown, radio).
  3. Check task logs for applied/skipped/failed counters.
  4. Review post-run summary and adjust thresholds.

After one stable run, scale to more pages and higher daily limits.

8. Troubleshooting

LLM test shows “connection failed”

  • Verify LM Studio server is running and model is loaded.
  • Use http://localhost:1234/v1.
  • Leave model field blank for auto-select or use exact model ID.

Captcha interruptions

  • Invisible reCAPTCHA: sign in to trusted account and retry.
  • hCaptcha: complete manually when prompted.

Easy Apply rate limit

  • If rate-limited and task is Easy Apply only, stop task and retry next day.
  • If external apply is enabled, skip easy applies and continue with supported external flows.

Form question validation failures

  • For numeric-only fields (like GPA), avoid text responses.
  • For dropdown/radio required fields, ensure defaults include location + authorization context.

9. Power User Workflow (Bot Operator Mode)

Borrowing from high-discipline automation playbooks (including sneaker-bot style ops):

  • Run small batches, validate quality, then scale volume.
  • Keep one known-good config profile as a fallback baseline.
  • Log unsupported domains and prioritize modules by frequency.
  • Rotate between high-quality keywords instead of broad scraping.
  • Review post-run outcomes daily and tune thresholds weekly.
Quality > quantity. Higher acceptance and fewer platform flags come from controlled pacing.

10. Support + Escalation

If you need help with a run, include these artifacts in your report:

  • Task ID and timestamp window
  • Relevant task logs
  • Target URL and platform
  • Screenshot of blocking UI state
